# Python 副本差距报告

本文档记录了当前仓库中的 Python 副本与原始 `pi` 编码代理（位于 `packages/coding-agent/`）之间的当前差距。

它的目标是作为后续逐步收敛工作的动态基线文档。

## 当前 Python 基线

当前 Python 实现已经包含：

- 通过可配置 `base_url` 接入 OpenAI 兼容提供方
- 流式输出助手响应
- 带工具结果回放的工具调用循环
- 受工作区范围限制的内置工具
- 本地 JSON 会话持久化
- 会话切换、分叉、重命名和保存
- 通过 `/reload` 手动重载运行时
- 通过 `/reload code` 尽力而为地重载模块
- 面向少量运行时选项的项目级/全局 YAML 配置

这意味着 Python 项目已经覆盖了较窄的“核心编码执行框架”路径，但尚未覆盖 `pi` 更广泛的产品能力面。

## 高层差距概览

### 已有近似实现

以下领域在两个项目中都存在，但 Python 版本的范围通常小得多：

- CLI 入口
- 有状态代理循环
- 流式文本响应
- 工具调用与工具结果续接
- 基础会话持久化
- 手动运行时重载
- 模型选择
- 可配置的提供方端点

### 仍然存在的主要差距

与原始 `pi` 相比，当前最大的剩余差异包括：

- 交互式 TUI 及全部终端 UI 组件
- RPC 模式和面向 SDK 的进程集成
- 提供方生态及认证流程
- 扩展、技能、提示模板和主题系统
- 更丰富的设置/资源加载与重载行为
- 更高级的会话管理与压缩
- 更广的命令面与工作流命令
- HTML 导出及相关辅助能力
- `pi` 包的包管理能力

## 详细差距

## 1. UI 与运行模式

原始 `pi` 支持多种运行模式，并提供完整的交互式终端 UI。

原始 `pi` 包含：

- 带自定义 TUI 渲染的交互模式
- Print 模式
- RPC 模式
- 面向 SDK 的集成接口
- 用于选择器、对话框、树视图、设置、主题和工具渲染的 UI 组件

Python 当前仅包含：

- 纯 REPL 风格 CLI
- 单次提示模式
- 最小化的流式控制台输出

Python 尚缺少：

- 差量终端渲染
- 会话选择器 UI
- 设置 UI
- 树形选择器 / 分支可视化
- 交互式编辑器组件
- 页脚 / 状态组件
- RPC 传输与命令协议
- Print/RPC 模式对齐

## 2. 提供方支持与认证

原始 `pi` 构建在 monorepo 的 AI/Provider 层之上，支持更广泛的提供方集合以及多种认证策略。

原始 `pi` 支持：

- Anthropic
- OpenAI
- Azure OpenAI
- Google Gemini
- Google Vertex
- Amazon Bedrock
- Mistral
- Groq
- 在更大项目中支持基于订阅 / OAuth 的提供方流程

Python 当前支持：

- 仅支持 OpenAI 兼容 API 形态
- 仅支持 API Key 认证
- 可在运行时覆盖 `base_url` 以接入内部网关

Python 尚缺少：

- 提供方注册表和提供方特定选项映射
- 类似 `/login` 的 OAuth / 订阅登录流程
- 按提供方自动加载模型目录
- 提供方能力检测
- 每个提供方的多传输方式选择
- 模型家族默认值与更丰富的解析逻辑

## 3. 配置、设置与重载架构

原始 `pi` 具有更广泛的设置与资源加载架构。

原始 `pi` 包含：

- 设置管理器
- 资源加载器
- 可重载的提示、上下文文件、技能和扩展
- 主题热重载
- 比简单运行时值更多维度的设置

Python 当前支持：

- 全局 YAML 配置：`~/.astra-python/config.yaml`
- 项目 YAML 配置：`.astra/config.yaml`
- 可重载的运行时配置项：
  - `model`
  - `base_url`
  - `system_prompt`
  - 工具启用列表
  - 已选工具默认值

Python 尚缺少：

- 自动文件监听
- 主题配置与主题热重载
- 上下文文件加载
- 资源加载器抽象
- 丰富的设置分类
- 超出简单文件之外的持久化认证 / 配置后端

说明：Python 中的 `/reload code` 只是开发便利能力，并不等同于原始项目中更广义的重载行为。

## 4. 工具与执行模型

Python 版本具备核心内置工具，但缺少更广泛的外围生态。

Python 当前包含：

- `read`
- `write`
- `edit`
- `ls`
- `find`
- `grep`
- `bash`

相较原始 `pi`，Python 缺少或弱化了：

- 由扩展提供的工具
- 工具包装与扩展钩子拦截
- UI 中更丰富的工具渲染
- 与提供方 / 模型相关的工具能力处理
- TUI 和扩展运行时所使用的更广泛外围工具能力

Python 的工具集是有意保持窄范围并限制在工作区内的；这是设计使然，但相对于原始项目的可扩展性仍然构成差距。

## 5. 会话、分支与压缩

原始 `pi` 拥有更高级的会话系统。

Python 当前包含：

- JSON 会话文件
- 会话保存 / 加载
- 重命名
- 切换
- 分叉

Python 尚缺少：

- 会话树导航 UI
- 分支摘要
- 自动压缩
- 手动 `/compact` 支持
- 更接近原始行为的上下文窗口管理策略
- 更丰富的会话元数据处理
- 更完整的恢复会话体验

在 UI / 提供方支持之外，这是最重要的功能差距之一。

## 6. 命令面

Python 当前仅支持少量斜杠命令。

当前 Python 命令集合：

- `/help`
- `/model`
- `/base-url`
- `/tools`
- `/sessions`
- `/switch`
- `/fork`
- `/rename`
- `/reload`
- `/reload code`
- `/save`
- `/exit`

原始 `pi` 暴露了更广的运行命令面，其中包括围绕认证、模型管理、设置、重载、压缩、会话等工作流的命令。

Python 至少缺少：

- `/login`
- `/compact`
- 更丰富的设置 / 模型管理命令
- 与主题、提示、包或扩展资源相关的命令
- 任意面向 RPC 的命令面

## 7. 可扩展系统

原始项目从设计上就是可扩展的。

原始 `pi` 提供以下一等概念：

- 扩展
- 技能
- 提示模板
- 主题
- Pi 包

Python 当前并不包含这些系统。

这意味着 Python 当前是一个独立执行框架，而不是一个可定制的平台。

## 8. 导出与辅助功能

原始 `pi` 在核心循环之外还包含额外的产品能力。

Python 尚缺少：

- HTML 导出
- 图像相关工具以及更丰富的多模态流程
- 与更大项目绑定的变更日志 / 帮助类工具
- 包安装 / 更新 / 删除 / 列表工作流
- 原始项目文档与实现中体现出的平台特定打磨

## 9. 验证与测试成熟度

Python 当前主要依赖有针对性的冒烟验证。

当前 Python 验证方式：

- `compileall`
- 本地冒烟脚本
- 交互式 CLI 冒烟检查

相比原始项目，Python 尚缺少：

- 更广泛的自动化测试套件结构
- 提供方交叉测试
- 在 abort / 上下文 / Unicode / 工具边界场景上的同等深度覆盖
- 针对会话与重载行为的回归覆盖

## 建议的收敛顺序

如果目标是选择性收敛而不是完全对齐，建议顺序为：

1. 会话与压缩能力改进
2. 超出 OpenAI 兼容层的提供方抽象
3. 更丰富的配置 / 资源加载
4. 更广的命令面
5. 扩展 / 技能 / 提示模板系统
6. RPC 模式
7. TUI 与主题系统

原因：

- 会话质量与提供方广度能最快提升核心可用性。
- 在实现有意义的可扩展性之前，需要先具备足够成熟的资源 / 配置体系。
- TUI 对齐成本高，应放在运行时架构更稳固之后。

## 非目标（除非明确提出）

不应默认将以下事项视为 Python 项目的自动目标：

- 与原始 `pi` 完全功能对齐
- 立即实现 TUI 对齐
- 在 Python 中复刻整个 monorepo 架构
- 复刻所有提供方 / 认证流程
- 一步到位复刻扩展 / 包生态

Python 项目应继续以渐进方式收敛，每新增一个子系统都应由用户价值和维护成本来证明其合理性。

## 更新指引

当 Python 项目获得新的子系统，或者实质性缩小了某项差距后，应按以下方式更新本文档：

- 将该能力从“缺失”移动到“已具备”或“部分具备”
- 描述精确的当前边界，而不是理想化的最终状态
- 除非行为确实可比，否则避免使用“已对齐”之类含糊表述

## 2026-03-08 更新

Python 项目现在还新增了一层最小 capability runtime。

新增为“已具备”或“部分具备”的能力：

- 用 registry 驱动的命令分派替代单一硬编码 CLI 命令树
- 超出单个 `system_prompt` 字符串的 prompt 组装
- 从 `.astra/prompts/*.md` 加载本地 prompt
- 从 `.astra/skills/*/skill.yaml` 加载本地 skill
- 通过 `/skill:<name>` 和 `/template:<name>` 在当前进程会话中激活能力
- 通过 `/runtime` 和 `/runtime warnings` 查看 runtime 状态
- 通过 `/runtime prompt` 和 `/runtime json prompt` 检查最终 prompt 及其组装顺序

当前 prompt inspection 聚焦于最终实际生效的 assembled prompt，以及真实参与组装的 fragment。

这缩小了哪些差距：

- 实质性缩小了核心可扩展基础设施方面的差距
- 部分缩小了命令面方面的差距
- 部分缩小了配置/资源加载方面的差距

即使在这次更新之后，仍然缺少：

- 跨进程重启持久化的 skill/template 激活状态
- 独立的 context source 抽象；当前 skill 的 context 文件仍然会折叠进 prompt 文本
- compaction strategy 与 `/compact`
- Python 代码扩展与 extension hook 管线
- 基于 provider capability 的 prompt/tool 条件启用
- 自动文件监听；资源变更仍需要手动 `/reload`
- 对 skipped fragment 的结构化诊断仍未落地；当前 prompt inspection 主要聚焦最终生效结果
